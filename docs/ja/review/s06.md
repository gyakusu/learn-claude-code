## 検証結果

ドキュメントの記述と実装コード (`agents/s06_context_compact.py`) を照合しました。

### 正確性: ほぼ正確、軽微な差異あり

**正確に書かれている点:**
- 3層構成（micro_compact → auto_compact → manual compact）の全体アーキテクチャ
- `THRESHOLD = 50000`、`KEEP_RECENT = 3` の定数値
- トランスクリプトを `.transcripts/` に保存する仕組み
- auto_compact の要約プロンプトとメッセージ置換ロジック
- ツール数は5つ（bash, read_file, write_file, edit_file, compact）

**軽微な差異:**
- ドキュメントの micro_compact 疑似コードでは `tool_name` 変数が未定義のまま使われている（L62）。実装では `tool_name_map` を構築して正しく解決している（L79-86）。ドキュメントは簡略化しすぎている
- ドキュメントの auto_compact の返却メッセージでは `transcript_path` が含まれないが、実装では含まれる（L119）。実装の方が実用的

**全体として、説明は本当のことを言っている。**

### 最も大事な学習ポイント

**1. 「3層の段階的積極性」という設計パターン**

これが最大のポイント。コンテキスト管理は一律ではなく、段階的に行う:

| 層 | いつ | コスト | 情報損失 |
|---|---|---|---|
| micro_compact | 毎ターン | ゼロ（API呼び出しなし） | 低（古いツール結果のみ） |
| auto_compact | 閾値超過時 | LLM 1回 | 中（要約に圧縮） |
| manual compact | モデルが判断 | LLM 1回 | 中 |

**安いものから先にやる。**高コストの圧縮は本当に必要なときだけ。

**2. 「何も失われない」原則**

`auto_compact` は圧縮前に必ず `.transcripts/` にフルログを保存する（L100-104）。これにより圧縮は可逆になる。アクティブコンテキストから外すだけで、情報を破壊しない。

**3. micro_compact の「ゼロコスト圧縮」**

LLM呼び出しなしで、古いツール結果をプレースホルダーに置換するだけ（L88-93）。ツール結果は往々にして巨大（ファイル全体など）なので、これだけでかなりのトークンを節約できる。**最もコスパの良い圧縮**。

**4. `compact` ツールでモデル自身に圧縮判断を委ねる**

第3層は「モデルが自分で compact ツールを呼ぶ」設計。閾値だけでなく、モデル自身が「今の文脈が散漫になった」と判断したときにも圧縮できる。これは Claude Code の `/compact` コマンドと同じ発想。

---

**一言でまとめると:** コンテキストウィンドウは有限だが、「安い順に3段階で圧縮＋フルログ保存」により、情報を失わずに無限セッションを実現する、というのがこの章の核心です。
