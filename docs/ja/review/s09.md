## 検証結果

ドキュメントと実装コード (`agents/s09_agent_teams.py`) を照合しました。

### ドキュメントは本当か？

**概ね正確**ですが、いくつか細かい差異があります。

| ドキュメントの記述 | 実装との一致 | 備考 |
|---|---|---|
| TeammateManager が config.json で名簿管理 | **正確** | |
| spawn() がスレッドでエージェントループ開始 | **正確** | ただし実装は既存メンバーの再spawn（idle/shutdown状態から）にも対応しており、ドキュメントより実装が賢い |
| MessageBus の send/read_inbox | **ほぼ正確** | ドキュメントの `read_inbox` は `"[]"` (文字列) を返すが、実装は `[]` (リスト) を返す。ドキュメントのコード断片は簡略化されている |
| ツール数 9 (+spawn/send/read_inbox) | **正確** | 実際は +spawn/send/read_inbox/list/broadcast の5つ追加で合計9ツール |
| `_teammate_loop` でインボックスを `<inbox>` タグで注入 | **差異あり** | ドキュメントではそう書いているが、実装のチームメイトループ(166行目)では `json.dumps(msg)` を直接 user メッセージとして追加し、`<inbox>` タグは使っていない。`<inbox>` タグを使うのはリードのループ(351行目)のみ |
| msg_type のバリデーション | **ドキュメントに記載なし** | 実装には `VALID_MSG_TYPES` による検証がある |

**結論**: ドキュメントは設計意図を正しく伝えていますが、コード断片は簡略化・理想化されており、実装の細部とは異なる箇所があります。学習資料としては問題ないレベルです。

### 最も大事な学習ポイント

**1. サブエージェント vs チームメイトの根本的な違い**
- s04のサブエージェントは「使い捨て」（spawn → 作業 → 要約返却 → 消滅）
- s09のチームメイトは「永続的」（spawn → working → idle → working → ...）で、アイデンティティと状態を持つ

**2. ファイルベースのメッセージバス（JSONL inbox）**
- 各チームメイトが自分専用の `.jsonl` ファイルを持ち、追記専用（append-only）で書き込み、読み取り時にドレイン（空にする）するパターン
- これにより**スレッド間の通信をファイルシステム経由で疎結合**に実現している
- 複雑なIPC機構を使わず、デバッグも容易

**3. アーキテクチャパターン: Lead + N Teammates**
- リード（メインループ）がチームメイトをspawnし、メッセージで指示を出す「オーケストレーター」パターン
- 各チームメイトは独立したエージェントループを持ち、自律的にツールを使える

**4. 実用上の注意点**
- 各チームメイトが独立してLLM APIを呼ぶため、**コストが並列に増加**する
- スレッド間の競合（config.json の同時書き込みなど）に対するロック機構がないため、本番利用には追加の工夫が必要

一言でまとめると: **「ファイルをメールボックスとして使い、独立したエージェントループをスレッドで走らせることで、最小限のインフラでマルチエージェント協調を実現する」**というのがs09の核心です。
