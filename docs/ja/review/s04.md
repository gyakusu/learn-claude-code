## レビュー結果

### 資料の正確性

ドキュメントの内容は**概ね正確**です。実装コード (`agents/s04_subagent.py`) と照合した結果を以下にまとめます。

**正確な記述:**
- サブエージェントが `messages=[]` （空のコンテキスト）で開始する点 → 実装L117で確認
- 子に `task` ツールを渡さず再帰的生成を防ぐ点 → L102-112で確認
- 最終テキストのみ親に返す点 → L134で確認
- 安全制限30回のループ → L118で確認
- 出力を50000文字で切り詰める点 → L131で確認
- `run_subagent()` のコード例 → 実装とほぼ一致

**軽微な差異:**
- ドキュメントでは基本ツール「5」と記載しているが、実装上の `CHILD_TOOLS` は **4つ** (`bash`, `read_file`, `write_file`, `edit_file`)。s03 には `list_files` があった可能性があるが、s04の実装では4つ。ここは若干不正確。
- ドキュメントのコード例では `task` ツールに `description` パラメータがないが、実装にはある（オプション）。これは省略による簡略化で問題なし。

---

### 最も大事な学習ポイント

**1. コンテキスト隔離こそが核心**

サブエージェントの最大の価値は「親のコンテキストを汚さない」こと。子が30回ツールを呼んでファイルを読みまくっても、親が受け取るのは1段落の要約だけ。これにより:
- 親のコンテキストウィンドウが節約される
- 親の思考が明晰に保たれる（ノイズが入らない）

**2. 構造は驚くほど単純**

サブエージェントは特別なAPIではない。単に **新しい `messages=[]` で同じエージェントループを回す関数** にすぎない。s03のループ構造を理解していれば、`run_subagent()` はその再利用。

**3. 再帰防止の設計パターン**

子に `task` ツールを渡さないことで無限再帰を防ぐ。これは `PARENT_TOOLS = CHILD_TOOLS + [task]` という差分で表現されている（L138）。シンプルだが重要な設計判断。

**4. ファイルシステムは共有、コンテキストは分離**

子は親と同じ作業ディレクトリにアクセスできる（ファイル読み書き可能）が、会話履歴は完全に独立。これは Claude Code の Agent ツールと同じ考え方。

---

**一言でまとめると:** サブエージェントとは「新しい messages 配列で同じループを回し、結果だけ返す関数」。特別な仕組みではなく、コンテキスト管理の問題をプロセス分離で解決するパターン。
